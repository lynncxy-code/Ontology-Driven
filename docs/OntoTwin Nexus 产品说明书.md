# OntoTwin Nexus 产品说明书

> 版本：3.0
> 定位：本体驱动的工业数字孪生平台
> 读者：使用者 / 新加入的协作者 / 部署到工厂的现场人员

---

## 1. 这是什么

OntoTwin Nexus 把工厂的 **CAD 图纸**和**业务数据**，变成"有类型、有坐标、有身份"的孪生实例，实时同步进 **UE5** 三维场景做监控。

它解决的根本问题是一句话：

> **画出来的形状 ≠ 业务上的身份。**

一张 CAD 图能告诉你"这里有个溶铜槽"，但说不出"这是溶铜槽 DW-007、挂着哪个传感器"。OntoTwin 把"形状"和"身份"分开处理，再焊接起来，让每个三维物件都对应一个真实可监控的设备。

---

## 2. 核心理念：把"一个东西"拆成三件事

3.0 的脊梁是一条四段流水线：

```
类型(是什么种类) → 构件(在哪、什么形状) → 实例(它是谁) → 运行(它怎么样)
```

| 层 | 回答什么 | 产物 | 在哪做 |
|---|---|---|---|
| 本体层 | 有哪些类型、什么关系 | 类型库（ObjectType + 接口 + 属性） | 语义图谱总览 / 本体配置中心 |
| CAD 解析层 | 图纸里哪里有个什么形状 | **构件**（匿名：类型 + 几何 + 坐标） | 坐标标定工作台 |
| 绑定层 | 这个构件是谁（实例编号） | **实例**（构件 + 业务身份） | 实例绑定台 |
| 运行层 | 实例此刻什么状态 | 实时监控 | 实例运维 / 监控台 / UE |

**关键术语链：类型 → 构件 → 实例**
- **类型**：一类东西的定义。
- **构件**：CAD 解析出的单个实物，有形状有坐标，**没有身份编号**。
- **实例**：构件绑定了业务实例编号后铸造而成，是唯一有资格被监控、被 UE 渲染的单元。

---

## 3. 功能模块（菜单：定义 → 建模 → 接入 → 运行）

### ① 数据定义
- **语义图谱总览**：可视化类型与关系的语义网络；类型库（数据集）的新建 / 合并 / 激活 —— **唯一建库入口**。
- **本体配置中心**：定义对象类型、绑定能力接口（如 I3D_Representable）、挂属性。

### ② 模型工坊
- **坐标标定工作台**：五步流水线
  1. 上传 DXF/DWG，解析设备块与图层
  2. 类型审核（勾选要纳入的设备块 / 线图层；只能合并进已有类型库）
  3. 锚点标定（4 点仿射变换，CAD 坐标 → UE 世界坐标，带 RMSE 校验）
  4. 构件列表（逐个核对，绑定 UE 资产路径）
  5. 导出 / **保存构件**到当前项目
- **CAD 一键成模**：程序化生成墙 / 地面。

### ③ 实例接入
- **实例绑定台**：对账双栏
  - 左：CAD 构件；右：业务实例清单（上传 CSV 或手填实例编号）
  - **自动撮合**：构件的 EQUIP_ID 精确匹配清单实例编号，一键绑定
  - **手选 / 手填**：未匹配的逐个挑号，或填清单外的新号（1:1 占用校验）
  - **铸造为实例**：绑定完成后铸造 —— 实例的唯一出生地
  - 墙体 / 地面等无身份构件不在此处

### ④ 运行守护
- **实例运维与监控**：实例列表、在线 / 离线、状态总览、override。
- **实时数据监控台**：实时位置脉动、事件流（SSE）、健康检查。

### ⑤ UE5 消费侧
- **OntoTwinSync 插件**：关卡放一个 TwinSceneManager，填后端地址 → Play 即自动 spawn 实例、轮询同步；**只渲染已绑定铸造的实例**。

---

## 4. 一条完整的使用路径（端到端）

1. **建类型库**：语义图谱总览 → 新建数据集（如"我的工厂 5F"）→ 激活。
2. **标定构件**：坐标标定 → 上传 DXF → 类型审核（合并进上面的类型库）→ 锚点标定 → 保存构件。
3. **绑定身份**：实例绑定台 → 上传实例清单 CSV（或手填）→ 自动撮合 + 手选 → 铸造为实例。
4. **运行监控**：实例运维 / 实时监控台查看；UE 放 TwinSceneManager → Play → 三维场景自动出现这些实例。

---

## 5. 数据底座

**统一 Project 模型**：一个工厂楼层 = 一个自洽项目。

```
Project {
  object_types     类型库
  components       构件（CAD 标定产出，匿名）
  instance_roster  业务实例清单（实例编号目录）
  instances        实例（绑定身份后铸造）
  calibration      坐标标定
}
```

- 一项目一文件、原子写、只加载激活项目 —— 根治"重启丢数据 / 实例串台"。
- 全局 `mapping_rules.json`：类型属性 → UE 渲染规则的翻译，跨项目共享。

---

## 6. 设计原则（约束）

- 单人开发，不过度工程化；不加异步队列 / 关系型数据库 / 权限系统。
- 前端走 CDN，不引入构建工具。
- 前端遵循 `ontotwin-ui` 规范：极简黑白灰、状态用 6px 点、无 emoji、无原生弹窗、操作可逆。
- 唯一可见性规则：前端 / UE / 投产，全部只认"当前激活项目"。

---

## 7. 部署提示（拿去工厂用）

- 后端 Flask + Docker；改了挂载代码用 `docker compose restart backend` 重启进程（`up --build` 不会重启）。
- 数据落在 `backend/data/projects/`（一项目一文件），备份 / 迁移一个工厂 = 拷一个文件。
- UE 端：关卡放 TwinSceneManager，填后端地址，留空 SceneId = 跟随后端当前激活项目。
