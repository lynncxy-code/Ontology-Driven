# OntoTwin 3.0 — 本体与 CAD 分层架构重构总纲 (PRD)

> 版本：3.0（架构总纲）
> 性质：方向性文档，建立共识。新页面的详细需求见配套《3.0.1 实例绑定台需求说明书》。
> 适用：单人开发，不过度工程化。

---

## 1. 背景与动机

当前产品把**一个被压扁的概念**当成了一件事：坐标标定工作台一旦标完类型与"实体"，就默认"实例已经产生"，可直接进入实例运维与监控。

但这是错的：

> **CAD 解析出来的东西是"匿名构件"——它有形状、有坐标、有类型，但没有业务身份（身份证号）。**

某些 CAD 设备从未与它在业务上的实例 ID 绑定，却被当作实例推进了运维侧。结果就是"有壳无魂"：场景里立着一个溶铜槽，但它不对应 MES/业务系统里任何一个真实设备，拉不到实时数据、挂不上传感器。

**3.0 的核心目标：把"画出来的形状"和"业务上的身份"这两件事彻底分开，并新增一个专门做身份绑定的环节。**

---

## 2. 核心重构：三层分离 + 术语链

把原先压扁的流程拆成三层，各司其职：

| 层 | 回答什么 | 产物 | 在哪做 |
|---|---|---|---|
| **本体层** | 有哪些**类型**、什么关系 | 类型库（ObjectType + 接口 + 属性） | 语义图谱总览 / 本体配置中心 |
| **CAD 解析层** | 图纸里**哪里**有个**什么形状** | **构件**（类型 + 几何 + 坐标，**匿名**） | 坐标标定工作台 |
| **映射绑定层** | 这个构件**是谁**（身份证号） | **实例**（构件 + 业务身份） | 实例绑定台（新增） |

### 统一术语链（全站对齐）

> **类型 → 构件 → 实例**
> 种类（ObjectType） → 匿名实物（Component） → 有身份的实物（Instance）

- **类型 / 对象类型（ObjectType）**：一类东西的定义。
- **构件（Component）**：CAD 解析出的单个实物，有类型、几何、坐标，**无身份证号**。取代原先工作台里"实体"的说法。
- **实例（Instance）**：构件绑定了业务身份证号后铸造而成，是孪生系统里唯一有资格被监控、被 UE 渲染的单元。

> 命名决策：原"实体（entity）"统一改称 **构件**。不用"对象"，因为会与"对象类型（ObjectType）"混淆。

---

## 3. 新数据流水线

**旧（3.0 之前）：**
```
CAD 上传 → 类型审核(可新建库) → 标定 → 实体 → 直接投产为"实例" → 实例运维 / UE
                                                  ▲
                                       问题：这些"实例"没有身份证号
```

**新（3.0）：**
```
本体层：  语义图谱总览 / 本体配置  ──建库/编排类型──►  类型库（唯一建库入口）
                                                          │
CAD 层：  坐标标定工作台  ──解析──► 构件  ──标定坐标──►  构件（挂到已有类型，只合并不建库）
                                                          │
绑定层：  实例绑定台  ──构件 ⨯ 实例清单(身份证号)──►  实例（唯一出生地）
                                                          │
运行层：  实例运维 / 监控 / UE  ◄── 只认已绑定实例 ──────┘
```

四个关键变化：
1. **坐标标定只产"构件"，不再产实例**（决策 Q1）。
2. **建库权限上收到本体层**：坐标标定的类型审核只能"合并到现有数据集"，不能"发布新数据集"。
3. **实例的唯一出生地是绑定台**：构件绑定身份证号后才铸造为实例。
4. **运行侧（实例运维 + UE）只认已绑定实例**（决策 Q3）。

## 4. 各模块变更清单

### 4.1 坐标标定工作台（`coord_workbench.html`）— 主要改动

| 项 | 变更 |
|---|---|
| 术语 | 第④步"实体列表" → **"构件列表"**；相关文案（实体/entity）全部改"构件" |
| 产物语义 | 标定完成的产物是**构件**（持久化到当前项目的 `components`），**不再 spawn 实例** |
| 投产按钮 | 原"投入实例库"语义改为"**保存构件**"（或"提交构件到项目"）；下游进绑定台，不直达实例运维 |
| 写入策略 | 类型审核的"写入策略"**去掉"发布为新数据集"**，仅保留"合并到现有工作数据集"（建库收口，见 4.2） |
| 已有作用域规则 | 保留 2.9.2.2 的图层筛选 + 类型审核勾选 + 线图层审核（构件作用域闸门不变） |

### 4.2 语义图谱总览 / 本体配置（`ontology_graph.html` / `ontology.html`）

**（a）Bug 修复：新建数据集预览图谱空白**
- 根因：新建数据集时后端写入的 `graph_data = {"nodes":[...], "links":[], "categories":[]}`（`app.py:314`）——`links` 与 `categories` 为空。前端图谱在节点无分类、无连线时渲染为空/散点，看不到"真正的图谱"。
- 处理：建库时正确填充 `categories`（按类型/图层分类）与必要的 `links`（类型—接口—属性关系），或在前端对空 categories 做兜底分类，保证预览有结构。

**（b）建库权限收口（决策 Change 2b）**
- **新建类型库只能在本体层完成**（语义图谱总览 / 本体配置）。
- 坐标标定的类型审核**只能合并**到已有数据集，不能新建。
- 语义价值：创建本体是本体层的职责；CAD 层只负责"把构件挂到已有类型上"，无建库权限。

### 4.3 接口治理 — 新增"实例绑定台"

- 在"接口治理"分组下新增页面（详见《3.0.1》）。
- 与现有**映射控制台**（`mapping.html`，做"类型属性 → UE 渲染规则"翻译）区分开：
  - 映射控制台 = **渲染规则映射**（属性→UE 表现）
  - 实例绑定台 = **身份绑定**（构件→身份证号→实例）
- 命名建议：用户初称"本体规则映射"，为避免与映射控制台混淆，建议页面正式命名为 **"实例绑定台"**（备选：构件入册台）。

### 4.4 实例运维 / 监控 / UE（`instance.html` / `floor_pulse.html` / OntoTwinSync）

- 只展示/渲染**已绑定实例**（决策 Q3）。纯几何构件（墙、地面等无身份对象）按既有几何通道处理，不进实例册。
- UE 快照接口 `/api/v2/state/snapshots` 返回的仍是当前激活项目的实例集——但实例集现在由绑定台铸造，天然都有身份证号。

## 5. 数据模型影响（ProjectStore）

当前 Project = `{object_types, instances, calibration, dataset}`。3.0 引入"构件"层后：

```
Project {
  object_types   类型库（不变）
  components     ★新增：CAD 标定产出的匿名构件 { component_id: {type, cad_xy, ue_xy, attribs(含 EQUIP_ID), asset_path, bound_instance_id?} }
  instances      实例（绑定身份后铸造）{ 身份证号: {component_id, object_type_rid, raw_state, ...} }
  instance_roster 业务实例清单（上传/手填的身份证号目录），可空
  calibration    标定（不变）
}
```

- **构件与实例分离存储**：构件是 CAD 事实，实例是业务事实，1:1 关联（决策 Q4）。
- 坐标标定写 `components`；绑定台读 `components` + `instance_roster`，写 `instances`。
- 详细字段在《3.0.1》定义。

## 8. 风险与迁移

- **改名波及面**：`coord_workbench.html` 内"实体"相关文案/变量较多，集中替换并 `node --check` + 浏览器全链路实测。
- **流水线断点**：标定不再直达实例运维，过渡期监控侧可能"空"，需在绑定台跑通后再验证运维侧。
- **ProjectStore 结构变更**：新增 `components` / `instance_roster` 字段，属于约定内扩展（向后兼容：旧项目无此字段时按空处理）。涉及 JSON 存储结构变化，开发前再向用户确认一次。
- 集中在分支开发，旧流程保留至绑定台联调通过。
