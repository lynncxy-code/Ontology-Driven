# OntoTwin 3.1 — 三维坐标体系统一（设计草案）

> 版本：3.1（草案）
> 性质：架构设计文档，决策已由 /grill-me 逐题敲定（见本文 §11）。待你过目后进入开发。
> 依赖：3.0 三层架构（类型 → 构件 → 实例）
> 适用：单人开发，不过度工程化。

---

## 1. 背景与目标

3.0 之前，UE 世界坐标**既是存储格式又是事实来源**。这导致：CAD 与 UE 顶视图朝向对不上（打点时要脑内旋转）、ontotwin 无法定义原点、多坐标源（海康等）无统一汇合点、坐标/状态映射"裸奔"——每接一个新源就临时定义一套规则。

**目标**：让 ontotwin 成为「**空间的事实来源**」——持有一个中立的**规范坐标系（canonical frame）**，任意外部坐标源通过"定义一帧 + 一个到规范系的变换"接入。

> 核心原则：不让任何一方的坐标当绝对真理。设一个中立世界帧，所有源各自声明"我→规范"的变换。接新源 = 填一帧参数，O(1)，互不影响。

---

## 2. 核心模型：规范坐标系 + 帧 + 变换链

```
【源坐标】各自原点/单位
  ├ CAD 帧 :  mm，图纸原点
  └ 外部帧 :  mm，外部系原点（海康等，本版只留接口）
        │   T_源→规范   （每源一个：2D 仿射，含单位归一）
        ▼
【规范坐标】建筑坐标系 · 单位 mm · 2D 平面 + 楼层 Z · 原点由 ontotwin 声明
        │   T_规范→UE   （每项目一个：显式声明的轴映射/翻转/旋转/原点 + ×0.1→cm）
        ▼
【UE 世界】cm · 实例绝对坐标（B1：UE 端 SetActorLocation 不变）
```

- **规范坐标是事实来源**；UE 厘米坐标是**派生缓存**（快照时算出，写进 `raw_state.translation_*`，UE/快照链路几乎不动）。
- **CAD 帧**是本版第一个具体帧；**外部帧**（海康等）本版只建通用抽象、不写专属逻辑。

## 3. 数据结构（ProjectStore 扩展）

> 涉及存储结构变更，已获用户同意（决策 §11.2）。向后兼容：旧项目无新字段时按空/默认处理；现有数据靠重导 CAD 迁移。

```jsonc
Project {
  // …现有：object_types / instances / calibration / dataset…

  "spatial_profile": {                 // ★项目空间剖面（规范系定义 + 规范→UE）
    "unit": "mm",
    "canonical_origin": [0, 0],        // 规范系原点（mm），ontotwin 声明
    "ue_transform": {                  // 规范 → UE 的显式声明（§5）
      "axis_map": { "x": "+x", "y": "-y" },   // CAD/规范轴 → UE 轴
      "flip": false,
      "rotation_deg": 0,               // 0/90/180/270 + 微调
      "ue_origin_cm": [0, 0],          // 规范原点落在 UE 世界的哪
      "scale_to_cm": 0.1               // mm→cm 固定
    },
    "floor_table": [                   // ★楼层 ↔ Z ↔ UE Level ↔ mapCode（L1）
      { "floor": 1, "z_base_mm": 0,    "ue_level": "W18_1F", "map_codes": [] },
      { "floor": 2, "z_base_mm": 4500, "ue_level": "W18_2F", "map_codes": [] }
    ]
  },

  "frames": [                          // ★通用帧注册表
    {
      "id": "frame_cad",
      "name": "CAD 图纸",
      "kind": "cad",                   // cad | external
      "unit": "mm",
      "to_canonical": {                // 源 → 规范 的仿射（§4）
        "method": "anchor",            // anchor | params
        "matrix": [[a,b,tx],[c,d,ty]]
      },
      "floor": 1,                      // 该帧所属楼层（可空）
      "map_code": null                 // 外部帧用（海康 mapCode）
    }
  ],

  "components": {
    "cmp_x": {
      "frame_id": "frame_cad",
      "canonical_xy": [x, y],          // ★规范坐标（mm）= 事实来源
      "floor": 1,
      "source_xy": [35008, 968861],    // 原始源坐标（留痕，可重算）
      "ue_xy_cache": [16.13, 29.88],   // 派生缓存（cm）
      // …object_type_rid / attribs / asset_path / bound_instance_id…
    }
  }

  // instances.raw_state.translation_x/y/z 仍为 UE cm，但为"派生缓存"，
  // 由 canonical + spatial_profile.ue_transform 算出
}
```

## 4. 变换公式

**源 → 规范**（每帧一个 2D 仿射，把源坐标统一进 mm 规范系）：
```
[x_canon]   [a b][x_src]   [tx]
[y_canon] = [c d][y_src] + [ty]
```
- CAD 帧：由锚点最小二乘拟合（沿用现有 `coord_transform`）。
- 外部帧：锚点拟合，或由源参数（原点偏移/尺度/旋转）直接构造。两种都编译成上面这个矩阵。

**规范 → UE**（每项目一个，显式声明，§5）：
```
1. 轴映射 + 翻转 + 旋转  → 2×2 朝向矩阵 R
2. p_ue_cm = R · p_canon_mm · 0.1 + ue_origin_cm
3. z_ue_cm = floor_table[floor].z_base_mm · 0.1
```
- 反算（UE → 规范 → 源）= 求各矩阵的逆，用于将来对外部系下发指令。

---

## 5. 规范→UE 朝向：显式声明 + 画布自动转向（本版关键 UX）

**问题**：CAD 是右手系（X 右 Y 上），UE 是左手系（Z 上、X 前 Y 右），两者差旋转 + 一次镜像。过去朝向"撞"在锚点拟合里，用户**打点时要脑内旋转**对应。

**方案**：把朝向做成**显式声明的参数**（`spatial_profile.ue_transform`：轴映射 + 翻转 + 旋转角 + 原点），不靠锚点撞。

**收益（直接消痛点）**：
- 标定画布**按声明自动转向显示** → 屏幕上的 CAD 与 UE 顶视图朝向一致，打点 1:1，不用脑内旋转。
- 改朝向参数**实时预览**：先粗选（90°/翻转）看画布转正，再放锚点精调 —— 破解"先有朝向才能打点、朝向又来自打点"的死循环。
- 改一处全场重算：朝向/原点对不上时，改这一个变换，所有实例瞬间重新派生 UE 坐标。

> 具体"CAD 哪个轴对 UE 哪个轴、是否翻转"是**每项目声明一次**的值，现场比一次填入。

## 6. 标定方式（源→规范，两种都支持）

| 方式 | 怎么做 | 适用 |
|---|---|---|
| 锚点拟合 | 源系/规范系各点 2~3 个同名已知点，最小二乘拟合仿射 | CAD（现状）；源不暴露参数时 |
| 直接填参 | 填原点偏移 + 尺度 + 旋转，构造仿射 | 源有文档化坐标系参数时 |

下游只认最终那个矩阵，不关心来源方式。

---

## 7. UE 侧（B1，不动 C++）

- 后端把"规范→UE"算完，下发 UE 绝对坐标；UE 端 `SetActorLocation` 一行不改。
- 顶视朝向修正在后端变换里完成，确定性、不碰编译。
- 单设备微调 = 在 ontotwin 改该实例的规范坐标数值（UE 端手动拖是临时的，会被轮询覆盖）。
- 根 Actor + 相对坐标（B2，整场拖动/多工厂拼关卡）**延后**到真需要时。

## 8. API 草案

| 方法 | 路径 | 作用 |
|---|---|---|
| GET/PUT | `/api/v2/spatial/profile` | 读/写当前项目空间剖面（单位/原点/规范→UE/楼层表） |
| GET/POST | `/api/v2/spatial/frames` | 列/建帧 |
| POST | `/api/v2/spatial/frames/{id}/calibrate` | 标定该帧的"源→规范"（锚点或填参） |
| POST | `/api/v2/spatial/preview` | 给一批源坐标，返回规范坐标 + UE 坐标（画布实时预览用） |
| —  | 现有 `/coord/*`、`/binding/*`、`/state/snapshots` | 内部改为：写规范坐标、快照时派生 UE 坐标 |

## 9. 本版边界（决策 §11.5）

**做**：项目空间剖面 · 规范坐标存储（UE 坐标派生）· CAD 帧接入 · 楼层→Z 表 · **通用帧抽象** · 单实例微调入口 · 标定画布按声明自动转向。

**不做（留接口/延后）**：海康等外部源的实时接入与专属逻辑 · 状态模型形式化 · 根 Actor（B2）· 白皮书/Skill 沉淀。

## 10. 分步实施计划

1. **后端数据结构**：ProjectStore 加 `spatial_profile` / `frames`；component 加 `canonical_xy/frame_id/floor`。
2. **变换层**：实现"源→规范""规范→UE"的矩阵构造与正/反算；快照派生 UE 坐标。
3. **CAD 帧接入**：坐标标定改为写规范坐标（锚点拟合 → frame.to_canonical）。
4. **空间剖面 UI + 画布自动转向**：朝向显式声明 + 实时预览。
5. **楼层表 + 单实例微调入口**。
6. **联调**：导入 → 标定（画布已按朝向转正）→ 绑定铸造 → UE 朝向正确、改一处全场重算。

> 每步可独立验证；前端改动前过 `ontotwin-ui` 规范。

## 11. 已决策记录（/grill-me 定稿，2026-06-23）

1. 规范单位 = mm；维度 = 2D + 楼层 Z。
2. 存储 = 规范坐标为事实来源，UE 坐标降为派生缓存。
3. UE 耦合 = B1；单设备微调在 ontotwin 改数值；B2 延后。
4. 楼层 = L1 统一建筑系 Z=楼层偏移；UE 每层一 Level。
5. 边界见 §9。
6. 源→规范标定 = 锚点 + 填参 两种，统一成仿射矩阵。
7. 规范→UE 朝向 = 显式声明；标定画布按声明自动转向显示。
