## 一、总览

3.0 系列目标：**让 Nexus 本体定义的场景一键导出为 Isaac Sim 可用的 USD 文件**。

需要三方协作：
- **后端**（Python）：USD 导出引擎、Scene API、资产转换
- **前端**（Nexus Web）：场景管理 UI、导出入口
- **UE 侧**：TwinSceneBuilder 改造、回写功能、导出入口按钮

**训练侧（Isaac Sim / Isaac Lab）由其他同事负责，不在本计划内。**

---

## 二、里程碑划分

### 🚩 M0：Spike 验证（3-5 天）

**目标**：在进入正式开发前，消除最大的技术不确定性。

**任务**：

| 任务 | 负责方 | 产出 |
|---|---|---|
| OpenUSD Python API 学习 | 后端 | 一个可跑通的 "hello cube" USD 生成脚本 |
| FBX → USD 转换工具选型 | 后端 | 选定 NVIDIA asset_converter 或 Blender，写出对比报告 |
| Isaac Sim 加载 USD 冒烟测试 | 训练侧支持 | 确认生成的 USD 能在 Isaac Sim 中正确显示 |
| UE 5.x USD Exporter 能力评估 | UE 侧 | 确认 UE 导入/导出 USD 的边界（评估是否需要自研）|
| 坐标系换算端到端验证 | 后端 + UE | 一个已知位置的 Cube 在 Nexus → USD → Isaac Sim → UE 各端显示一致 |

**退出标准**：
- 能生成一个含 5 个物体的 USD，Isaac Sim 能加载、看到物体、应用物理
- 有明确的技术栈选型结论
- 所有团队成员对 OpenUSD 的关键概念（Stage / Layer / Prim / Reference / customData）有基本理解

**风险**：如果 Spike 阶段发现 OpenUSD 与 Isaac Sim 的某个核心能力有兼容问题，需要重新评估整体方案。

---

### 🚩 M1：后端核心链路打通（2 周）

**目标**：后端能把一个写死的 Scene 数据导出为 USD 文件。

**任务**：

| 任务 | 模块 | 工时估算 |
|---|---|---|
| 数据库 Schema 迁移（新增 scenes / scene_exports 表）| 3.1 | 1 天 |
| Scene CRUD API 实现 | 3.1 | 3 天 |
| Instance.belongs_to_scenes 字段改造与迁移脚本 | 3.1 | 2 天 |
| I3D_PhysicsHint 接口定义与 API 支持 | 3.0 | 1 天 |
| FBX → USD 缓存式转换服务 | 3.0 F2 | 3 天 |
| Scene → USD 核心导出引擎 | 3.0 F1 | 5 天 |
| DXF → USD 结构生成（复用 ezdxf 逻辑） | 3.0 F3 | 3 天 |
| 单元测试 + 导出报告生成 | 3.0 F7 | 2 天 |

**产出**：

- 可运行的 REST 接口：`POST /api/v3/scenes/{scene_id}/export`
- 能生成带 customData 追溯信息的 USD 文件
- 能支持 static / dynamic / graspable 三种物理类型
- 能处理 DXF 结构（墙/地板/柱子）

**退出标准**：
- 以 Postman 触发导出，能得到 Isaac Sim 可加载的 USD
- 场景复杂度：10 实例以上，含 graspable 物体
- customData 完整，训练侧能通过 instanceId 反查本体

**里程碑演示**：端到端演示"本体数据 → 后端 API → USD → Isaac Sim 加载"的完整链路。

### 🚩 M4：联调与稳定（1 周）

**目标**：前后端 + UE + 训练侧端到端联调，解决集成问题。

**任务**：

- 三端联调测试（后端 / 前端 / UE）
- 与训练侧协作：确认 Isaac Lab 能正确加载导出的 USD
- 性能优化（大场景、批量导出）
- 文档完善（用户手册 + 运维手册）
- 回归测试 2.x 功能（确认没破坏原有能力）
- Bug 修复

**退出标准**：
- 一个含 100+ Instance 的真实场景能完整跑通：本体配置 → 导出 → 训练
- 训练侧反馈可用，至少能跑 Isaac Lab 的一个抓取 demo
- 核心 API 的 P99 响应时间达到 3.0 文档中定义的性能指标

## 四、依赖与风险管理

### 4.1 外部依赖

| 依赖 | 风险等级 | 应对 |
|---|---|---|
| OpenUSD Python SDK（usd-core）| 低 | 稳定库，团队先做 Spike |
| NVIDIA asset_converter | 中 | 若不可用，切换 Blender 备选方案 |
| Isaac Sim 环境可用 | 中 | 训练侧需要同期配齐环境，M0 必须验证 |
| UE 5.x USD 支持 | 低 | 已验证可用，但对某些 Schema 的支持有限 |

### 4.2 关键风险

| 风险 | 级别 | 影响 | 应对 |
|---|---|---|---|
| 团队 OpenUSD 学习曲线 | **高** | 进度延期 | M0 必须完成 Spike，团队内部分享培训 |
| FBX 转 USD 材质损失 | 中 | 视觉效果降级 | 接受降级，后续版本通过 MDL 补偿 |
| 坐标系转换 Bug | 中 | 物体位置错位 | 写单元测试；端到端验证 |
| graspable 资产的碰撞几何质量 | **高** | 抓取训练失败 | 资产库管理员同步准备专用 physics_proxy |
| UE 侧 HTTP 调用不稳定 | 低 | 偶发加载失败 | 重试机制 + 本地缓存 |
| 数据库 Schema 迁移阻塞 | 中 | 前端等待 | 迁移脚本写在 M1 最前面 |

### 4.3 并行化策略

为压缩总周期，以下工作可并行：

- **M0 的 Spike** 可由后端和 UE 并行进行
- **M1 的后端** 和 **M2 的前端** 可在 M1 后期并行（前端 mock API 先开发）
- **M3 的 UE 侧** 可在 M1 完成后立即启动（不用等 M2 完成）

**最短路径**：M0 (5 天) → M1 (10 天) → M2 & M3 并行 (10 天) → M4 (5 天) = **30 天**
